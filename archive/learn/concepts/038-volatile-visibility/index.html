Volatile / visibility

A volatile field tells Java that reads must see the latest write across threads, so it is mainly a visibility guarantee.
It is good for publication and stop flags, but it does not make compound actions like count++ atomic.
This is not Java's answer to STM: when many writes must change together, Java still uses locks or atomics, while Loom tries to reduce the need for shared mutable state in the first place.

// Java
class Demo {
  static volatile boolean stop = false;

  public static void main(String[] args) throws Exception {
    // Start a virtual thread that keeps reading the shared stop flag.
    Thread worker = Thread.startVirtualThread(() -> {
      while (!stop) {
        Thread.onSpinWait();
      }
      // Once the main thread writes stop = true, this read sees it and exits.
      System.out.println("stopped");
    });

    // Let the worker run for a moment, then publish the stop signal.
    Thread.sleep(100);
    stop = true;   // visible to the worker thread
    // Wait until the worker observes the flag and finishes.
    worker.join();
  }
}
