F1: Too Many Arguments

Functions should have a small number of arguments — none is best, then one, then two; three is already a smell, and more than three needs a very special justification.
A long parameter list is hard to read and easy to misuse: callers must remember the exact order, and two adjacent values of the same type can be swapped without the compiler ever noticing.
The fix is usually to notice that several arguments travel together and belong to a concept of their own — wrap them in an object whose fields are named, so the call site reads clearly and the order no longer matters.

Avoid — a wall of positional parameters
// Java
public void createUser(String firstName, String lastName, String email,
                       String street, String city, String zip, String country,
                       boolean newsletter, boolean admin) {
    // ...
}

// Call site: which boolean is which? is it city then zip, or zip then city?
createUser("Ada", "Lovelace", "ada@x.io",
           "5 Analytical Ave", "London", "EC1", "UK", true, false);
Prefer — group related arguments into named objects
// Java
public void createUser(Name name, Address address, UserOptions options) {
    // ...
}

// Call site: every value is labelled by the field it fills.
createUser(
    new Name("Ada", "Lovelace"),
    new Address("5 Analytical Ave", "London", "EC1", "UK"),
    new UserOptions(/* newsletter */ true, /* admin */ false));
