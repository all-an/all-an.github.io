E1: Build Requires More Than One Step

Building a project should be a single, trivial operation — one command that takes a clean checkout all the way to a finished artifact.
When the build instead demands a sequence of manual steps — sync the submodules, copy a config file, regenerate sources, all in exactly the right order — every step is a chance to get it wrong, and a newcomer burns an afternoon of tribal knowledge before the first successful compile.
The fix is to fold the whole sequence into one checked-in script or build target, so anyone can go from clone to build with a single command.

Avoid — a checklist of manual steps in the project wiki
# Shell — "How to build", copied from the team wiki
git submodule update --init                              # 1. pull vendored deps by hand
cp config/local.example.properties config/local.properties  # 2. or it won't start
./scripts/gen-version.sh > src/main/resources/version.txt   # 3. forget this and the build fails
(cd vendor/protobuf && make)                             # 4. compile the proto toolchain
mvn -o package                                           # 5. ...and finally build
Prefer — one checked-in command that does it all
# Shell
./build.sh   # syncs deps, generates sources, compiles, and tests — one step, clean checkout to artifact
