F2: Output Arguments

We expect arguments to be inputs to a function and the return value to be its output — that is how we read code without a second thought.
An output argument breaks that expectation: the function reaches into a parameter and mutates it to hand a result back, so the reader has to check the signature and the body to discover that appendFooter(report) actually changes report.
The fix is to return the result instead, or — if the function is really acting on an object — to make it a method on that object so the change of state is expected.

Avoid — a parameter that is secretly an output
// Java
// Does this read the report, or modify it? You can't tell from the call.
public void appendFooter(StringBuilder report) {
    report.append("\n-- generated ").append(LocalDate.now());
}

StringBuilder report = buildReport();
appendFooter(report);   // surprise: `report` was mutated
Prefer — make state changes belong to the object
// Java
// `this` is the obvious thing being changed; no hidden output parameter.
public class Report {
    private final StringBuilder body = new StringBuilder();

    public void appendFooter() {
        body.append("\n-- generated ").append(LocalDate.now());
    }
}

Report report = buildReport();
report.appendFooter();   // clearly acts on `report`
