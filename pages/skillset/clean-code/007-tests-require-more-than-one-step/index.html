E2: Tests Require More Than One Step

Running the tests should be a single, obvious command — one that works straight from a clean checkout with nothing set up by hand.
When the suite instead needs a database started, a schema loaded, environment variables exported, and a fixture script run first, the tests become a chore people skip: a suite that is hard to run is a suite that rarely runs, and untested code rots quietly.
The fix is to make the command spin up everything it needs and tear it down again, so a teammate — or CI — can verify the whole project with one button.

Avoid — a setup ritual before the tests will even start
# Shell — "How to run the tests", from the README
docker run -d -p 5432:5432 postgres:16        # 1. start a database by hand
export DATABASE_URL=postgres://localhost/test  # 2. point the app at it
psql "$DATABASE_URL" -f db/schema.sql           # 3. load the schema
./scripts/seed-fixtures.sh                       # 4. insert the fixtures
mvn -o test                                      # 5. ...now run the tests
Prefer — one command that provisions, tests, and cleans up
# Shell
./test.sh   # starts an ephemeral DB, migrates, seeds, runs the suite, then tears it all down
