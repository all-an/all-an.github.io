Scoped value / dynamic context

A scoped value binds immutable context to a bounded execution scope so deep code can read it without threading parameters through every method.
Loom pairs ScopedValue with structured concurrency so child tasks inherit the same read-only context cleanly, which is usually a better fit than mutable ThreadLocal state for virtual-thread code.
This still is not STM: Java is not making shared updates transactional here, it is making context propagation explicit, bounded, and safer so less mutable shared state is needed.

// Java
import java.lang.ScopedValue;
import java.util.concurrent.StructuredTaskScope;

final class Demo {
  static final ScopedValue<String> REQUEST_ID = ScopedValue.newInstance();
}

// Bind request context for this whole block and for child tasks created inside it.
ScopedValue.runWhere(Demo.REQUEST_ID, "req-42", () -> {
  try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
    // The child task inherits the scoped value and can read it without a method parameter.
    var task = scope.fork(() -> "handling " + Demo.REQUEST_ID.get());
    // The parent waits for the child before the scoped binding goes away.
    scope.join();
    scope.throwIfFailed(RuntimeException::new);
    // Control returns here with the child result, still inside the bounded scope.
    System.out.println(task.get()); // handling req-42
  } catch (InterruptedException e) {
    Thread.currentThread().interrupt();
  }
});
