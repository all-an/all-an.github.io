Dependency injection

Dependency injection (DI) means a class receives the things it depends on from the outside instead of creating them itself. This is a form of Inversion of Control: the decision of which concrete dependency to use is taken away from the class and given to whoever assembles the program. A class that depends on an interface and accepts an implementation through its constructor is decoupled from any concrete type — and, crucially, becomes testable, because a test can inject a fake.
Frameworks automate the wiring — Spring IoC and CDI (Java), Dagger / Hilt (Android), ASP.NET Core DI (C#), Angular DI (TypeScript), ZLayer (Scala ZIO), the Reader monad (Haskell/Scala) — but the principle needs no framework at all, just a few constructor calls in one place. It was named by Martin Fowler / Robert C. Martin. This is a runnable Maven project; see README.md.

First, the anti-pattern: the class builds its own dependency, hard-wiring itself to one concrete type. There is no seam, so it cannot be tested without sending real email:

// ANTI-PATTERN — do NOT do this. The dependency is created inside the class.
class OrderService {
  private final MessageSender sender = new EmailSender();  // hard-wired!

  void placeOrder(String customer, String item) {
    sender.send(customer, "Your order for " + item + " is confirmed.");
  }
}
// A test of placeOrder() would now fire a REAL EmailSender. There is no way to
// substitute a fake — the dependency is welded in.

Now with constructor injection: the dependency is an interface, passed in. The class no longer knows or cares which implementation it got:

// The dependency abstraction — the "seam" we can substitute.
interface MessageSender {
  void send(String recipient, String message);
}

class OrderService {
  // The injected dependency — chosen by the caller, not by this class.
  private final MessageSender sender;

  // Constructor injection: the dependency arrives as a parameter.
  OrderService(MessageSender sender) {
    this.sender = sender;
  }

  void placeOrder(String customer, String item) {
    sender.send(customer, "Your order for " + item + " is confirmed.");
  }
}

The concrete wiring happens once, at the composition root — the entry point where the object graph is assembled. A DI framework automates exactly this step:

public class Main {
  public static void main(String[] args) {
    MessageSender sender = new EmailSender();        // choose the real one...
    OrderService orders = new OrderService(sender);  // ...and inject it.
    orders.placeOrder("ada@example.com", "Analytical Engine");
  }
}

The payoff is testability. Because the dependency comes from outside, a test injects a fake that records calls instead of sending email, then asserts on it — no network, no real mail server:

// A test double implementing the same interface — it records instead of sends.
static class RecordingSender implements MessageSender {
  final List<String> sent = new ArrayList<>();
  public void send(String recipient, String message) {
    sent.add(recipient + " | " + message);
  }
}

@Test
void placingAnOrderNotifiesTheCustomer() {
  RecordingSender fake = new RecordingSender();
  OrderService orders = new OrderService(fake);   // inject the fake, not email

  orders.placeOrder("ada@example.com", "Analytical Engine");

  assertEquals(1, fake.sent.size());
  assertEquals(
      "ada@example.com | Your order for Analytical Engine is confirmed.",
      fake.sent.get(0));
}
