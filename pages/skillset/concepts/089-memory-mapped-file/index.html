Memory-mapped file

Normally a program reads a file by copying bytes out with read() and writes by copying them back with write() — each call crosses into the kernel. A memory-mapped file removes the copying: FileChannel.map() asks the OS to back a region of the process's address space with the file itself (mmap), so from then on the file's bytes are memory. Touching a byte whose page is not resident faults it in from disk on demand; a modified page is written back lazily by the OS. Because the mapping is shared, every mapping of the same file refers to the same physical pages — so two mappings act as shared memory, and edits survive as the file. Here a MappedFile reads and writes ints and strings at byte offsets through one MappedByteBuffer.
MappedByteBuffer (Java); mmap (C, Python); memmap2 (Rust); MemoryMappedFile (C#). This is a runnable Maven project; see the README.md in this concept's folder in the cloned repository.

The file is opened as a channel and a region of it is mapped read/write. The returned MappedByteBuffer is just memory: a putInt is a plain memory write, not a syscall, and the OS flushes the dirty page to the file on its own schedule. Mapping a region larger than the file grows the file to fit:

FileChannel channel = FileChannel.open(path, CREATE, READ, WRITE);
MappedByteBuffer buffer = channel.map(FileChannel.MapMode.READ_WRITE, 0, size);

buffer.putInt(64, 41);       // a plain memory write — no read()/write() syscall
int value = buffer.getInt(64);   // faults the page in from disk if not resident

Values are addressed by byte offset rather than by stream position, so the mapped region is true random-access memory — you can jump anywhere and read or write in place:

public void putInt(int offset, int value) { buffer.putInt(offset, value); }
public int  getInt(int offset)            { return buffer.getInt(offset); }

Writes through the map are writes to the file, so the bytes persist. A brand-new mapping opened later — even after the first is closed — reads back the same data, because it went to the file, not to a transient stream buffer. force() (msync) flushes dirty pages to make them durable now rather than whenever the OS chooses:

try (MappedFile writer = new MappedFile(path, 128)) {
  writer.putString(0, "durable");
  writer.putInt(64, 99);
  writer.flush();                 // force dirty pages out to disk
}
try (MappedFile reader = new MappedFile(path, 128)) {
  reader.getString(0);            // "durable" — read back from a fresh mapping
  reader.getInt(64);             // 99
}

Because the mapping is shared, two mappings of the same file point at the same physical pages. A write through one is visible through the other with no read from disk, and no flush, in between — this is exactly what makes mmap a shared-memory mechanism and the basis of copy-free inter-process communication:

try (MappedFile a = new MappedFile(path, 128);
     MappedFile b = new MappedFile(path, 128)) {
  a.putInt(0, 12345);
  b.getInt(0);                    // == 12345 — seen through the other mapping at once
}

The tests pin the three defining properties — random-access memory, persistence across mappings, and shared memory between mappings:

@Test
void persistsAcrossRemappings() throws IOException {
  try (MappedFile writer = new MappedFile(path, 128)) {
    writer.putString(0, "durable");
    writer.flush();
  }
  try (MappedFile reader = new MappedFile(path, 128)) {
    assertEquals("durable", reader.getString(0));
  }
}

@Test
void sharesMemoryBetweenTwoMappings() throws IOException {
  try (MappedFile a = new MappedFile(path, 128);
       MappedFile b = new MappedFile(path, 128)) {
    a.putInt(0, 12345);
    assertEquals(12345, b.getInt(0));   // no disk I/O, no flush — shared pages
  }
}

Why bother: there is no per-call copy between kernel and user buffers, and the OS page cache is shared rather than duplicated — so databases, search indexes, and log stores map their files for speed. Only touched pages are loaded, so a file larger than RAM can be processed a page at a time by demand paging. And two processes mapping the same file share those pages, the classic copy-free IPC channel. The trade-off: mappings are page-aligned and fixed in size (growing the file means remapping), I/O errors surface as SIGBUS/faults rather than checked exceptions, and on the JVM the region is unmapped only when the buffer is garbage-collected — there is no explicit unmap.
