Decorator / wrapper

The Decorator pattern adds behavior to an object by wrapping it in another object that implements the same interface. Because the wrapper is the same type as what it wraps, callers cannot tell the two apart — and wrappers can wrap other wrappers, so behavior composes in layers chosen at runtime. It is the flexible alternative to subclassing: instead of an explosion of SugaryMilkyEspresso classes for every combination, you stack small, single-purpose decorators.
It is a Gang of Four structural pattern. The same idea appears as Python/TypeScript @decorators, Spring AOP Proxy objects, Rust newtype wrapper structs, HTTP middleware (Go, Express), and AOP aspects. Java's own I/O streams are decorators — new BufferedReader(new InputStreamReader(in)). This is a runnable Maven project; see README.md.

All participants share one interface — base component and decorators alike. That is what lets a wrapper stand in for what it wraps:

interface Coffee {
  int costCents();
  String description();
}

// The base component — what sits at the center of the wrapping.
class Espresso implements Coffee {
  public int costCents()      { return 200; }
  public String description() { return "Espresso"; }
}

The abstract decorator both implements Coffee and holds a Coffee. By default it forwards every call to the wrapped object; subclasses add to that result:

abstract class CoffeeDecorator implements Coffee {
  protected final Coffee inner;            // the next layer inward
  protected CoffeeDecorator(Coffee inner) { this.inner = inner; }

  // Default: delegate straight through. Subclasses override to add on top.
  public int costCents()      { return inner.costCents(); }
  public String description() { return inner.description(); }
}

class Milk extends CoffeeDecorator {
  Milk(Coffee inner) { super(inner); }
  // Take the inner result and ADD to it — never replace it.
  public int costCents()      { return inner.costCents() + 50; }
  public String description() { return inner.description() + " + Milk"; }
}

At the call site, wrapping reads from the inside out, and because each layer is itself a Coffee, the layers stack:

Coffee plain  = new Espresso();                          // Espresso = 200
Coffee latte  = new Milk(new Espresso());                // Espresso + Milk = 250
Coffee deluxe = new Sugar(new Milk(new Espresso()));     // + Milk + Sugar = 275

The test confirms the composition: each layer adds its contribution, the layers stack to any depth, and a decorated coffee is still a Coffee (so it can be wrapped again):

@Test
void decoratorsStackToAnyDepth() {
  Coffee coffee = new Sugar(new Milk(new Espresso()));

  assertEquals(275, coffee.costCents());                 // 200 + 50 + 25
  assertEquals("Espresso + Milk + Sugar", coffee.description());
}
