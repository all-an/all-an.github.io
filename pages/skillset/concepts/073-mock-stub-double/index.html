Mock / stub / double

A test double replaces a real collaborator so a test runs fast and in isolation — no network, database, or payment API. The class under test takes its collaborator through the constructor, so a test can hand it a double instead of the real thing.
Two kinds, from Meszaros' taxonomy. A stub returns canned answers and the test checks the result (state verification). A mock also records how it was called, so the test checks the interaction — was the collaborator used correctly? Below, the stub drives checkout to "confirmed", while the mock asserts the gateway was charged exactly once with the right account and amount.
This is hand-built with no framework so the distinction is explicit; production Java uses Mockito (and jest.fn(), unittest.mock, ScalaMock elsewhere) to generate doubles automatically.

// Java
// The real collaborator we do NOT want to call in a test — a live payment API.
interface PaymentGateway {
  boolean charge(String account, int cents);
}

// A STUB: returns a canned answer with no logic, to drive the code under test
// down a chosen path. The test then checks the RESULT (state verification).
static final class ApprovingStub implements PaymentGateway {
  public boolean charge(String account, int cents) {
    return true; // canned response — always approves
  }
}

// A MOCK: like a stub, but it also RECORDS how it was called, so the test can
// verify the INTERACTION (behavior verification), not just the result.
static final class RecordingMock implements PaymentGateway {
  int callCount = 0;
  String lastAccount;
  int lastCents;

  public boolean charge(String account, int cents) {
    callCount++;
    lastAccount = account;
    lastCents = cents;
    return true;
  }
}

// Mock assertion: verify the gateway was charged exactly once, with the right
// account and amount — catching bugs a stub cannot, like charging twice or $0.
RecordingMock mock = new RecordingMock();
new OrderService(mock).checkout("acc-2", 1299);
check("mock: charged exactly once", mock.callCount == 1);
check("mock: charged the right amount", mock.lastCents == 1299);

In real Java you would not hand-write those doubles — Mockito generates them. The same two tests with Mockito: when(...).thenReturn(...) stubs a canned answer, and verify(...) checks the interaction.

// Java + Mockito
import static org.mockito.Mockito.*;

// One generated double stands in for the real gateway — no class to write.
PaymentGateway gateway = mock(PaymentGateway.class);

// STUB (state verification): declare the canned answer, then assert the result.
when(gateway.charge("acc-1", 500)).thenReturn(true);
assertEquals("confirmed", new OrderService(gateway).checkout("acc-1", 500));

// MOCK (behavior verification): run the code, then verify HOW the gateway was
// used — charged exactly once, with the right account and amount.
new OrderService(gateway).checkout("acc-2", 1299);
verify(gateway, times(1)).charge("acc-2", 1299);
