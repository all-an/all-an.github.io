Repository / DAO

A Repository (or DAO, Data Access Object) is a collection-like interface for accessing domain objects. It names what the application can do with stored data — save, find, list, delete — while hiding how and where that data lives. The business code talks to the interface; the storage mechanism (a map, a SQL database, an HTTP API) stays behind it. Because callers depend only on the abstraction, the storage can be swapped with no change to the code that uses it — the Dependency Inversion Principle applied to persistence.
It is a Domain-Driven Design building block; the DAO form comes from the J2EE Core Patterns. In practice it appears as Spring Data's JpaRepository, Entity Framework's Repository<T>, Rails' Active Record, or Scala's Quill / Slick. This is a runnable Maven project; see the README.md in this concept's folder in the cloned repository.

The domain object is a plain, immutable value — an id plus its data, and no persistence code at all:

public record User(String id, String name, String email) {}

The repository interface is the data-access contract. It deals in domain objects, never in rows or SQL. findById returns an Optional so "not found" is an explicit, type-checked outcome:

public interface UserRepository {
  User save(User user);                       // insert OR update (by id)
  Optional<User> findById(String id);         // explicit "maybe absent"
  List<User> findAll();
  boolean deleteById(String id);              // true if one was removed
}

One concrete implementation, backed by a map. It is interchangeable with any other UserRepository — a JDBC or HTTP version — because callers depend on the interface, not on this class:

public class InMemoryUserRepository implements UserRepository {
  // The "table": id -> user. LinkedHashMap keeps insertion order.
  private final Map<String, User> usersById = new LinkedHashMap<>();

  public User save(User user) {
    usersById.put(user.id(), user);           // put = insert or replace
    return user;
  }
  public Optional<User> findById(String id) {
    return Optional.ofNullable(usersById.get(id));
  }
  public List<User> findAll() {
    return new ArrayList<>(usersById.values()); // a copy callers can't mutate
  }
  public boolean deleteById(String id) {
    return usersById.remove(id) != null;       // null means nothing was there
  }
}

All the consuming code talks only to the interface. The concrete class is named exactly once — on the line that creates it. save is modelled on a collection's put: the same call inserts a new user or updates an existing one matched by id:

// The only mention of the concrete class — swap this one line for a database.
UserRepository users = new InMemoryUserRepository();

users.save(new User("ada-1", "Alice", "alice@example.com"));   // INSERT
users.save(new User("bob-2", "Bob",   "bob@example.com"));

users.findAll();                       // [Alice, Bob] in insertion order
users.findById("ada-1");               // Optional[Alice]

users.save(new User("ada-1", "Alice", "alice@corp.com"));      // UPDATE, same id
users.deleteById("bob-2");             // true
users.findById("bob-2");               // Optional.empty()

The test field is typed as the interface, not the concrete class — so the very same tests would pass against a database-backed implementation:

private UserRepository users;

@BeforeEach
void setUp() { users = new InMemoryUserRepository(); }   // fresh, empty each test

@Test
void saveWithExistingIdUpdatesInPlace() {
  users.save(new User("ada-1", "Alice", "alice@example.com"));
  users.save(new User("ada-1", "Alice", "alice@corp.com"));   // same id

  assertEquals(1, users.findAll().size());                    // not duplicated
  assertEquals("alice@corp.com", users.findById("ada-1").orElseThrow().email());
}

Because the repository owns its storage, findAll hands back a copy: tampering with the returned list cannot corrupt the repository's internal state:

@Test
void findAllReturnsACopyThatCannotMutateTheRepository() {
  users.save(new User("ada-1", "Alice", "alice@example.com"));

  users.findAll().clear();                 // tamper with the returned list

  assertEquals(1, users.findAll().size()); // repository is unaffected
}

Repository vs. DAO: the terms overlap. A DAO typically maps one-to-one to a table and is framed around CRUD; a Repository is a domain concept that behaves like an in-memory collection of aggregate roots and may sit on top of one or more DAOs. Both share the core move shown here: hide the persistence mechanism behind an interface of domain operations.
