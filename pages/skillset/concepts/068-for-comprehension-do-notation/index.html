For comprehension / do notation

JavaScript's async/await is syntactic sugar over monadic flatMap/bind: it lets a chain of dependent steps read top-to-bottom instead of nesting. await is the Promise's flatMap, the code after it is the continuation, and a rejected step short-circuits the rest.
This shows both forms side by side: the async/await sugar and the nested .then chain it compiles to. They produce the same answer, and the unknown id #9 short-circuits both — proving async/await is the do-notation for Promises.
Scala's for comprehension and Haskell's do notation are the same sugar generalized: they work over any monad (Option, List, Either, IO), not just the async one. Java has no equivalent syntax: you either chain flatMap by hand (see Monad / chain) or use virtual threads, which solve the async case by a different mechanism rather than by desugaring binds.

// TypeScript
// Each lookup returns a Promise — the monad we are sequencing over.
function findUser(id: number): Promise<string> {
  const name = users[id];
  return name ? Promise.resolve(name) : Promise.reject(new Error(`no user #${id}`));
}
function findOrders(name: string): Promise<string[]> {
  return Promise.resolve(orders[name] ?? []);
}

// DO-NOTATION form: each `await` is a monadic bind. The steps read as plain
// sequential statements, yet `items` depends on `name` from the previous bind.
async function summarySugar(id: number): Promise<string> {
  const name = await findUser(id);        // bind 1
  const items = await findOrders(name);   // bind 2, depends on bind 1
  return `${name} ordered ${items.join(", ")}`;
}

// DESUGARED form: exactly what async/await compiles to — nested Promise.then,
// which is the Promise monad's flatMap. This is the bind chain the sugar hides.
function summaryDesugared(id: number): Promise<string> {
  return findUser(id).then((name) =>
    findOrders(name).then((items) =>
      `${name} ordered ${items.join(", ")}`));
}
