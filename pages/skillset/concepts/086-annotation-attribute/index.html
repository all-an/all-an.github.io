Annotation / attribute

An annotation (a C# attribute) is declarative metadata attached to a program element — a class, field, method, or parameter. It carries no behaviour of its own: it is a label a separate processor reads and acts on. The element states what should be true; the processor decides what to do about it. Here two custom annotations — @NotBlank and @Min — declare validation rules on a User's fields, and a tiny Validator reflects over the object at runtime to enforce them. This is precisely how Bean Validation, JPA, and JSON mappers work on your classes.
@Annotation (Java); [Attribute] (C#); #[derive] (Rust); @decorator (Python, TypeScript); @inline / @specialized (Scala); and pragma (C, D). This is a runnable Maven project; see the README.md in this concept's folder in the cloned repository.

A custom annotation is declared with @interface. Its members look like methods and become the parameters callers supply; a member may have a default. The crucial part is RUNTIME retention — annotations default to CLASS retention, which the JVM drops at load time, so without this the processor could never see it:

@Retention(RetentionPolicy.RUNTIME)   // keep it readable at runtime, not just in source
@Target(ElementType.FIELD)            // it may only annotate fields
public @interface NotBlank {
  String message() default "must not be blank";   // a member with a default value
}

An annotation can carry a parameter. @Min(18) configures its rule declaratively; the member named value lets callers omit the key:

@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.FIELD)
public @interface Min {
  int value();   // the inclusive lower bound; `value` allows the shorthand @Min(18)
}

The domain object is decorated with annotations but contains no checking logic. It declares what should be true and leaves enforcement to whatever processor reads the metadata:

public class User {

  // @NotBlank declares the rule; it does not enforce it. The Validator does.
  @NotBlank(message = "name is required")
  private final String name;

  // @Min(18) configures the rule with a parameter read straight off the metadata.
  @Min(18)
  private final int age;

  public User(String name, int age) { /* ... */ }
}

The processor is where the annotations finally mean something. It walks any object's fields, asks each one which rule annotations it carries, reads the field's value, and turns every violated rule into a message:

public static List<String> validate(Object target) {
  List<String> violations = new ArrayList<>();

  for (Field field : target.getClass().getDeclaredFields()) {
    field.setAccessible(true);   // read the value even though the field is private
    Object value = readValue(field, target);

    // @NotBlank: the String value must be present and non-whitespace.
    if (field.isAnnotationPresent(NotBlank.class) && isBlank(value)) {
      violations.add(field.getName() + ": " + field.getAnnotation(NotBlank.class).message());
    }

    // @Min: the int value must be at least the annotation's `value` parameter.
    if (field.isAnnotationPresent(Min.class)) {
      int minimum = field.getAnnotation(Min.class).value();   // read the parameter
      if (((int) value) < minimum) {
        violations.add(field.getName() + ": must be at least " + minimum);
      }
    }
  }
  return violations;
}

Putting it together: the same Validator validates any annotated object. A well-formed user yields no messages; an invalid one yields exactly one message per broken rule — without User or Validator hard-coding the other's details:

// Every rule the annotations declare is satisfied → no violations.
Validator.validate(new User("Ada", 36));     // []

// Blank name violates @NotBlank; under-age value violates @Min.
Validator.validate(new User("  ", 16));
// [name: name is required, age: must be at least 18]

The tests pin each annotation to its effect — the custom message, the @Min parameter, and that multiple broken rules each contribute one message:

@Test
void validUserHasNoViolations() {
  assertTrue(Validator.validate(new User("Ada", 36)).isEmpty());
}

@Test
void blankNameReportsTheAnnotationMessage() {
  assertEquals(List.of("name: name is required"), Validator.validate(new User("  ", 36)));
}

@Test
void ageBelowMinimumReportsTheBound() {
  assertEquals(List.of("age: must be at least 18"), Validator.validate(new User("Ada", 16)));
}

@Test
void everyViolatedRuleContributesOneMessage() {
  List<String> violations = Validator.validate(new User("", 10));
  assertEquals(2, violations.size());
}

Why bother: annotations let you state intent declaratively and keep the enforcing logic in one reusable place — Bean Validation (@NotNull, @Size), ORMs (@Column, @Id), and frameworks (@Test, @Autowired) all hand metadata to a tool that acts on it. The trade-off mirrors reflection's: the rule and its enforcement are decoupled, so a typo'd member or a missing processor fails at runtime, and the indirection stays invisible until you find the code that reads the annotation.
