Memoization

Memoization caches a function's result by its arguments, so a repeated call returns the stored answer instead of recomputing it. It only works for pure functions — same input must always mean same output — and trades memory for speed.
This builds a generic memoize wrapper with the standard library: ConcurrentHashMap.computeIfAbsent runs the real function only on a cache miss; on a hit it returns the stored value and the wrapped function never runs. Because memoize takes and returns a Function, it wraps any pure function without changing its callers.
A counter proves the saving: six calls over three distinct inputs run the real computation just three times — the other three are served from the cache. The same idea is what Python's @lru_cache, Guava/Caffeine caches, and dynamic-programming tabling all do.

// Java
// The expensive function we want to avoid repeating: a pure computation that
// bumps a counter every time it actually runs.
static int slowSquare(int n) {
  realCalls.incrementAndGet();
  return n * n;
}

// memoize wraps any function in a cache. computeIfAbsent runs fn only when the
// key is missing; on a hit it returns the stored value and fn never runs.
static <A, B> Function<A, B> memoize(Function<A, B> fn) {
  Map<A, B> cache = new ConcurrentHashMap<>();
  return arg -> cache.computeIfAbsent(arg, fn);
}

Function<Integer, Integer> fastSquare = memoize(Main::slowSquare);

// Six calls but only three distinct inputs (4, 4, 7, 4, 7, 9). The repeats are
// served from the cache, so the real function runs just three times.
int[] inputs = {4, 4, 7, 4, 7, 9};
for (int n : inputs) {
  System.out.println("square(" + n + ") = " + fastSquare.apply(n));
}
System.out.println("real computations: " + realCalls.get()); // 3, not 6
