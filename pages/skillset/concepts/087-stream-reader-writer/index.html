Stream / reader-writer

A stream is a sequence of bytes (or characters) you process a piece at a time rather than holding all at once. A reader is a source you pull from; a writer is a sink you push to. Decoupling them behind the InputStream / OutputStream abstractions means the same code works whether the bytes come from a file, a socket, or memory — and the payload can be far larger than RAM. Here a StreamCopier pumps bytes from a source to a sink through a small buffer, and an UpperCaseStream filter stage transforms bytes as they flow through the pipe.
InputStream / OutputStream (Java); io.Reader / io.Writer (Go); Read / Write trait and BufReader (Rust); file streams (C++, Python); fs2.Stream (Scala). This is a runnable Maven project; see the README.md in this concept's folder in the cloned repository.

The heart of the abstraction is a loop that moves bytes through a fixed-size window, not all at once. It reads a chunk, writes exactly the chunk that was read, and repeats until the source signals end-of-stream by returning -1. Memory stays bounded by the buffer no matter how large the stream is:

public static long copy(InputStream source, OutputStream sink) throws IOException {
  byte[] buffer = new byte[8 * 1024];   // an 8 KiB window, not the whole payload
  long total = 0;

  int read;
  while ((read = source.read(buffer)) != -1) {   // -1 means the source is drained
    sink.write(buffer, 0, read);                 // write only what was actually read
    total += read;
  }
  return total;
}

Nothing in that loop knows what the source or sink concretely is. A 14-byte string and a 14-gigabyte file copy with the identical code, because they meet only at the InputStream / OutputStream surface — that is the whole point of streaming over a sink, socket, or buffer.

A processing stage is itself an InputStream, so it drops into the same copy() with no special handling — streams compose like a Unix pipeline, cat file | tr a-z A-Z | …. This UpperCaseStream wraps an upstream source and uppercases each byte on the way through:

public final class UpperCaseStream extends FilterInputStream {

  public UpperCaseStream(InputStream source) { super(source); }

  @Override
  public int read(byte[] buffer, int offset, int length) throws IOException {
    int read = in.read(buffer, offset, length);   // pull from upstream
    for (int i = offset; i < offset + read; i++) {
      buffer[i] = (byte) Character.toUpperCase(buffer[i]);   // transform on the way through
    }
    return read;   // -1 when drained makes the loop a no-op and stops the copy
  }
}

Putting it together: the same StreamCopier moves bytes either straight through or routed through the filter stage — the source and sink here are in-memory buffers, but a file or socket would plug into the identical call:

// Plain copy: bytes flow source → buffer → sink unchanged.
StreamCopier.copy(source, sink);             // sink now holds "hello, streams"

// Filtered copy: an UpperCaseStream sits in the pipe and transforms each byte.
StreamCopier.copy(new UpperCaseStream(piped), upper);   // upper holds "HELLO, STREAMS"

The tests pin the core behaviour: every byte reaches the sink with the right count, an empty source copies nothing, a payload larger than the buffer still copies correctly (proving the chunked loop), and the filter transforms bytes while still plugging into the same copier:

@Test
void copiesEveryByteAndReportsTheCount() throws IOException {
  byte[] data = "hello, streams".getBytes(UTF_8);
  ByteArrayOutputStream sink = new ByteArrayOutputStream();
  long copied = StreamCopier.copy(new ByteArrayInputStream(data), sink);
  assertEquals(data.length, copied);
  assertArrayEquals(data, sink.toByteArray());
}

@Test
void payloadLargerThanBufferCopiesCorrectly() throws IOException {
  byte[] data = new byte[100_000];   // ~12× the internal 8 KiB buffer
  // ... fill, copy, and assert every byte arrived
}

@Test
void uppercasesEachByteFlowingThrough() throws IOException {
  ByteArrayInputStream source = new ByteArrayInputStream("hello, streams".getBytes(UTF_8));
  ByteArrayOutputStream sink = new ByteArrayOutputStream();
  StreamCopier.copy(new UpperCaseStream(source), sink);
  assertEquals("HELLO, STREAMS", sink.toString(UTF_8));
}

Why bother: streaming keeps memory bounded by the buffer, not the payload, so copying a file or piping a response never materialises the whole thing, and independent stages — buffering, compression, encryption, encoding (GZIPInputStream, CipherInputStream, BufferedReader) — snap together by wrapping one another. The trade-off: a stream is forward-only and stateful — you can read it once, you must close it to release the underlying resource, and a byte stream needs a charset before it becomes text.
