Non-blocking I/O

Blocking I/O parks a thread inside read() until data arrives, so each connection needs its own thread. Non-blocking I/O never waits: a call with no data ready returns immediately, and the thread instead asks the OS which channels are ready and services only those — so one thread multiplexes many connections. The multiplexer is a Selector (backed by epoll on Linux, kqueue on macOS/BSD, io_uring on newer Linux). Here a single-threaded NonBlockingEchoServer serves several clients through one select loop — the Reactor pattern.
NIO / Selector (Java); epoll (Linux), kqueue (BSD/macOS), io_uring (Linux 5.1+); asyncio (Python); tokio::io (Rust); http4s / blaze (Scala). This is a runnable Maven project; see the README.md in this concept's folder in the cloned repository.

The listening channel is set non-blocking and registered with the selector for ACCEPT readiness. The loop then blocks once on the whole set of channels — not on any single one — and is handed back exactly the channels that can make progress without waiting:

serverChannel.configureBlocking(false);                   // never block on this channel
serverChannel.register(selector, SelectionKey.OP_ACCEPT); // wake me when a client connects

while (running) {
  selector.select(100);   // wait up to 100 ms for ANY channel to be ready
  for (SelectionKey key : selector.selectedKeys()) {
    if (key.isAcceptable())     acceptClient();   // a connection is waiting
    else if (key.isReadable())  echo(key);        // a client has bytes ready
  }
}

When the listening channel is acceptable, accept() returns immediately. The new client channel is itself set non-blocking and registered for reads, so its future data wakes the same loop — no thread is dedicated to it:

SocketChannel client = serverChannel.accept();   // ready, so this returns at once
client.configureBlocking(false);
client.register(selector, SelectionKey.OP_READ); // wake me when this client sends data

Reading is the same story: the channel is only serviced once the selector says it is readable, so read() returns now rather than blocking. A return of -1 means the peer closed, so the key is cancelled and the loop carries on serving everyone else:

int read = client.read(buffer);   // ready, so this returns now — never blocks
if (read == -1) {                 // -1 means the peer closed the connection
  key.cancel();
  client.close();
  return;
}
buffer.flip();
client.write(buffer);             // echo the bytes straight back

Putting it together: one server thread handles every client. The clients themselves are ordinary blocking sockets — the multiplexing all happens on the server side:

NonBlockingEchoServer server = new NonBlockingEchoServer();
new Thread(server).start();   // a single thread runs the whole select loop

// Two clients, one server thread — both served by the same loop.
EchoClient.roundTrip(server.port(), "hello");        // → "hello"
EchoClient.roundTrip(server.port(), "non-blocking"); // → "non-blocking"

The tests pin the defining property — that a single loop echoes a client, multiplexes several clients with no thread each, and keeps serving after one disconnects:

@Test
void servesManyClientsOnTheSameThread() throws IOException {
  assertEquals("one",   EchoClient.roundTrip(server.port(), "one"));
  assertEquals("two",   EchoClient.roundTrip(server.port(), "two"));
  assertEquals("three", EchoClient.roundTrip(server.port(), "three"));
}

@Test
void keepsServingAfterAClientDisconnects() throws IOException {
  EchoClient.roundTrip(server.port(), "first");   // connects and disconnects
  assertEquals("second", EchoClient.roundTrip(server.port(), "second"));
}

Why bother: one thread can hold tens of thousands of mostly-idle connections, because a parked thread (≈1 MB of stack) is replaced by a cheap selector registration — this is the classic C10k problem, and how web servers, proxies, and databases scale. It is also the foundation of async/await, event loops (Node.js), and reactive runtimes (tokio, Netty, http4s). The trade-off: control flow is inverted into event handling, partial reads and writes must be handled by hand, and a single slow handler stalls every connection sharing the loop.
