Why D?

D is a systems programming language with the feel of a high-level language. It gives you manual memory control when you need it, a garbage collector when you don't, the most expressive metaprogramming system in any compiled language, and inline assembly syntax that actually makes sense. This lesson explains why D is an ideal language for learning the core principles of computer science — and why the skills you build here transfer directly to the languages the industry uses most.

Reason 01

Transferable Knowledge

D's syntax is deliberately close to C, C++, and Java — the three languages that dominate systems programming, enterprise software, and computer science curricula. If you learn D, picking up any of those languages feels like reading a dialect you already speak. Curly-brace blocks, familiar control flow, class and struct keywords, operator overloading — none of it requires unlearning.

More importantly, D forces you to understand the concepts that those languages implement in messier ways. Memory ownership, value vs reference semantics, type parameterisation, and compile-time evaluation are all first-class ideas in D, expressed with far less ceremony than in C++ and far fewer hidden rules than in Java. When you later read C++ templates or Java generics, you will already know what the compiler is trying to do — because D showed you the clean version first.

C / C++

Same curly-brace syntax, same pointer model, same struct/class split — but D's version has no header files and no undefined behaviour traps.

Java / C#

Classes, interfaces, modules, and garbage collection all exist in D. The concepts map directly; only the verbosity is gone.

Python / JS

D's foreach, ranges, and UFCS (uniform function call syntax) produce the same clean iteration style you know from scripting languages.

Rust

Ownership, lifetimes, and @safe memory guarantees — D covers the same ideas with more escape hatches, making the concepts easier to learn incrementally.

d — familiar syntax, clean semantics
// Everything here looks like C++ or Java — but compiles instantly
import std.stdio;

struct Point {
    double x, y;

    double distanceTo(Point other) const {
        import std.math : sqrt;
        return sqrt((x - other.x)^^2 + (y - other.y)^^2);
    }
}

void main() {
    auto a = Point(0, 0);
    auto b = Point(3, 4);
    writeln(a.distanceTo(b));   // 5.0
}
Reason 02

Optional Garbage Collector — True Systems Programming

Most languages force a binary choice: either you manage memory yourself (C, C++, Rust) or a garbage collector manages it for you (Java, Go, Python). D refuses that choice. The GC is on by default, which means you can write high-level code quickly and safely. But you can opt out of the GC at any granularity — a single function, a whole module, or the entire program.

Mark a function @nogc and the compiler will refuse to compile any GC-allocating code inside it. Use malloc / free directly, or D's own allocator library (std.experimental.allocator), and you have full manual control — with no runtime overhead. This is how D powers embedded systems, game engines, and OS kernels while also being used for scripting and tooling in the same codebase.

For learning, this is invaluable. You can start with the GC and focus on algorithms and data structures, then turn it off for specific sections to learn memory layout, stack vs heap allocation, and cache performance — all in the same language, without switching environments.

d — gc on and off in the same program
import core.memory : GC;
import core.stdc.stdlib : malloc, free;

// This function may use the GC freely
int[] buildList(int n) {
    int[] result;
    foreach (i; 0 .. n) result ~= i;   // GC allocation ✓
    return result;
}

// This function must never allocate through the GC
@nogc void criticalPath(int* buf, size_t len) {
    foreach (i; 0 .. len) buf[i] *= 2;   // no allocation — fine
    // buf ~= 1;  // ← compile error: GC forbidden here
}

void main() {
    auto list = buildList(10);

    // Manual allocation — pointer arithmetic, no GC involved
    int* raw = cast(int*) malloc(10 * int.sizeof);
    criticalPath(raw, 10);
    free(raw);
}
Reason 03

The Best Inline Assembly of Any Language

Every compiled language eventually offers a way to drop down to assembly. C and C++ use __asm__ with GCC's cryptic constraint syntax — a string-based interface that is error-prone and nearly impossible to read. Rust delegates to an asm! macro with its own mini-language. D has a different philosophy: inline assembly is a proper first-class sub-language with its own grammar, parsed and type-checked by the compiler itself.

In D, you write assembly inside an asm { } block using clean, Intel-style mnemonics. You can reference D variables and functions by name — no register constraints, no clobber lists, no opaque string literals. The compiler knows what you are doing and can diagnose mistakes at compile time.

This matters for learning because understanding how high-level constructs map to machine instructions is one of the deepest insights in computer science. D makes that exploration approachable for the first time.

d — readable inline assembly, no constraint strings
// Compute absolute value using x86-64 instructions
long absVal(long x) {
    asm {
        mov  RAX, x;      // load argument — D variable by name
        mov  RCX, RAX;
        neg  RCX;
        cmovl RAX, RCX;   // conditional move: pick positive version
    }
}

// CPUID — read processor features directly from hardware
uint cpuidMax() {
    asm {
        xor  EAX, EAX;    // leaf 0
        cpuid;
        // EAX now holds the maximum CPUID leaf
    }
}

// Compare: C requires this for the same result
// long absVal(long x) {
//     long r;
//     __asm__ ("movq %1, %%rax; movq %%rax, %%rcx; negq %%rcx;"
//              "cmovlq %%rcx, %%rax; movq %%rax, %0"
//              : "=r"(r) : "r"(x) : "rax", "rcx");
// }
Why it matters: Writing even a few assembly routines teaches you registers, calling conventions, flags, and the cost of every operation. D's clean syntax removes the accidental complexity so you can focus on the essential complexity — how the CPU actually works.
Reason 04

Metaprogramming — Code That Writes Code

Metaprogramming is the ability to inspect, generate, and transform code at compile time. It is the foundation of serialisation libraries, query builders, dependency injection frameworks, and test scaffolding in every major language. D has the most complete metaprogramming system of any compiled language, built around four interlocking features.

Templates

D templates are like C++ templates, but they work on values, types, and aliases equally, support constraints via if clauses, and produce error messages that actually identify the problem. A template can accept any combination of type parameters, value parameters, and alias parameters in the same parameter list.

d — constrained templates
import std.traits : isNumeric;

// Works for any numeric type; rejected at compile time for strings, etc.
T clamp(T)(T val, T lo, T hi) if (isNumeric!T) {
    return val < lo ? lo : val > hi ? hi : val;
}

void main() {
    assert(clamp(15, 0, 10) == 10);
    assert(clamp(3.5, 0.0, 5.0) == 3.5);
    // clamp("hi", "a", "z");  // ← compile error: string is not numeric
}

Compile-Time Function Execution (CTFE)

Any regular D function can be evaluated at compile time simply by using its result where the compiler needs a constant. No separate template specialisation, no constexpr restrictions — the same function works at runtime and at compile time. This lets you pre-compute lookup tables, validate configuration, and generate string-based code — all with zero runtime cost.

d — compile-time function execution
// Ordinary function — runs at runtime or compile time
pure int factorial(int n) {
    return n <= 1 ? 1 : n * factorial(n - 1);
}

// Evaluated at compile time — lives in read-only data segment
static immutable int[13] factTable = () {
    int[13] t;
    foreach (i; 0 .. 13) t[i] = factorial(i);
    return t;
}();

void main() {
    assert(factTable[10] == 3628800);   // no runtime work at all
}

Mixins

String mixins let you paste a string as D source code at compile time. Template mixins inject a template body into any scope. Together they are the mechanism behind D's serialisation libraries, ORM frameworks, and code generation tools — powerful enough to replace entire categories of reflection that Java and C# handle with runtime overhead.

d — string mixin generating property accessors
import std.format : format;

// Generates a private field + public getter/setter pair at compile time
string makeProperty(string type, string name)() {
    return format!(
        "private %s _%s;"
        "@property %s %s() const { return _%s; }"
        "@property void %s(%s v) { _%s = v; }"
    )(type, name, type, name, name, name, type, name);
}

struct Person {
    mixin(makeProperty!("string", "name"));
    mixin(makeProperty!("int",    "age"));
}

void main() {
    Person p;
    p.name = "Alice";
    p.age  = 30;
    assert(p.name == "Alice");
}

__traits and static foreach

The __traits keyword exposes the compiler's internal knowledge about any symbol at compile time: its type, its fields, its methods, its attributes. Combined with static foreach, you can iterate over all fields of a struct and generate serialisation code, validation logic, or debug output — without any runtime reflection and without any external framework.

d — compile-time struct introspection
import std.stdio;
import std.conv : to;

// Prints every field name and value of any struct at compile time
void dump(T)(const T obj) {
    static foreach (field; __traits(allMembers, T)) {
        writeln(field, " = ", __traits(getMember, obj, field).to!string);
    }
}

struct Config {
    string host = "localhost";
    int    port = 8080;
    bool   tls  = false;
}

void main() {
    dump(Config());
    // host = localhost
    // port = 8080
    // tls  = false
}
Reason 05

Design by Contract

Bertrand Meyer's Design by Contract is one of the most important ideas in software engineering: every function should specify what it requires of its caller (preconditions), what it guarantees it will produce (postconditions), and what invariants always hold on an object (class invariants). Most languages bolt this on through libraries or conventions. D builds it directly into the grammar.

d — built-in pre/postconditions and class invariants
struct BankAccount {
    private double _balance;

    // Invariant: balance must never be negative
    invariant { assert(_balance >= 0, "balance went negative"); }

    void deposit(double amount)
    in  { assert(amount > 0, "deposit must be positive"); }
    out { assert(_balance > 0, "balance must be positive after deposit"); }
    do  { _balance += amount; }

    void withdraw(double amount)
    in  { assert(amount > 0 && amount <= _balance, "invalid withdrawal"); }
    do  { _balance -= amount; }

    @property double balance() const { return _balance; }
}

void main() {
    auto acc = BankAccount();
    acc.deposit(100);
    acc.withdraw(30);
    assert(acc.balance == 70);
    // acc.withdraw(500);  // ← AssertError at runtime in debug builds
}
Learning value: Writing contracts forces you to think precisely about what your functions actually do. It is the closest code can get to a mathematical proof of correctness. This habit of thought transfers to every language and every codebase you will work with in your career.
Reason 06

Ranges — A Unified Abstraction for Sequences

In most languages, iteration is tied to concrete data structures: arrays have for loops, linked lists have iterators, and lazy sequences have their own protocol. D unifies all of these with the range concept. A range is any type that satisfies a small interface: empty, front, and popFront. Everything in std.algorithm and std.range works on any range — whether it is a slice of an array, a file read line by line, a mathematical sequence, or a live network stream.

The composability of ranges teaches you one of the deepest ideas in programming: separating what data is from how you process it. That principle underlies Unix pipes, SQL queries, functional programming, and modern stream APIs in every language.

d — lazy range pipeline, no intermediate allocations
import std.stdio;
import std.range     : iota, take;
import std.algorithm : filter, map, sum;

void main() {
    // Sum of squares of the first 10 odd numbers — no array created
    auto result =
        iota(1, 1000)                   // lazy: 1, 2, 3 …
        .filter!(n => n % 2 != 0)    // lazy: keep odd
        .take(10)                       // lazy: first 10
        .map!(n => n * n)             // lazy: square each
        .sum;                           // evaluate the whole pipeline

    writeln(result);   // 1+9+25+49+81+121+169+225+289+361 = 1330
}
Reason 07

A Graduated Safety System

D gives you three levels of memory safety that you can apply per-function and mix freely in the same file:

@safe

Compiler-enforced. No pointer arithmetic, no manual casts, no @system calls. Memory safety is guaranteed by the type system.

@trusted

You vouch for the safety of this function. Useful for wrapping C library calls that are safe to use but use raw pointers internally.

@system

Default. Full power: pointer arithmetic, manual memory, inline assembly. Explicitly unsafe, so the boundaries are always visible.

This model teaches you to think about safety boundaries explicitly. Rather than pretending unsafe operations do not exist (Java) or forcing all code into a single safety level (Rust), D shows you how to isolate unsafe code behind verified interfaces — the technique that every real-world systems project uses.

d — mixing safety levels
import core.stdc.string : memcpy;

// @safe — compiler checks every operation
@safe void safeSort(int[] data) {
    import std.algorithm : sort;
    data.sort();   // fine — sort is @safe
    // data.ptr++;   // ← compile error: pointer arithmetic forbidden
}

// @trusted — unsafe internally, but the interface is safe
@trusted void fastCopy(int[] dst, const int[] src)
in { assert(dst.length >= src.length); }
do  { memcpy(dst.ptr, src.ptr, src.length * int.sizeof); }

@safe void main() {
    int[] a = [3, 1, 4];
    int[] b = new int[3];
    fastCopy(b, a);   // safe to call from @safe context
    safeSort(b);
}
Reason 08

The Best Language for Learning Computer Science Fundamentals

Most languages make a pedagogical trade-off: they either abstract away the machine (Python, JavaScript) or expose so much complexity that beginners drown (C, C++). D sits precisely at the level where the machine is visible but not overwhelming.

Concept C C++ Java Python Rust Go D

Compile-time computation ✗ none ~ constexpr ✗ none ✗ none ~ const eval ✗ none ✓ full CTFE

D covers every layer of the computing stack — from machine instructions and memory allocators, through data structures and algorithms, up to type systems and language design. It does all of this with a single coherent set of rules, a single toolchain, and a single standard library. You never have to switch languages to learn the next concept. That is why it is the ideal language for this curriculum.

The goal of this curriculum is not to make you a D programmer. It is to use D as a lens through which every fundamental concept of computer science is visible, precise, and directly executable. By the end, you will understand not just how to write software, but why it works the way it does — at every level of abstraction from silicon to system design.
